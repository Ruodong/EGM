## Architecture

### i18n Pattern

```typescript
// Usage in any component:
const { t, locale, setLocale } = useLocale();
return <h1>{t('dashboard.title')}</h1>;
```

### SSR Hydration Safety

```typescript
// Always start with DEFAULT_LOCALE to match SSR output
const [locale, setLocaleState] = useState<Locale>(DEFAULT_LOCALE);

// Hydrate from localStorage after mount (client-only)
useEffect(() => {
  const saved = localStorage.getItem(STORAGE_KEY);
  if (saved === 'en' || saved === 'zh') setLocaleState(saved);
}, []);
```

This prevents hydration mismatches between server (no localStorage) and client.

### Dictionary Structure

Flat key-value maps with dot-notation grouping:
```typescript
// en.ts
export const en: Record<string, string> = {
  'dashboard.title': 'Dashboard',
  'dashboard.totalRequests': 'Total Requests',
  'nav.requests': 'Requests',
  // ...
};
```

### Translation Lookup

```typescript
const t = (key: string, fallback?: string): string => {
  return dictionaries[locale][key] ?? fallback ?? key;
};
```

Falls back to: dictionary value → explicit fallback → key itself.

## Notes

- Chinese content for questionnaire templates is stored in the database (questionTextZh, optionsZh, etc.) and treated as equal-status — not optional fallbacks
- The i18n system is frontend-only; backend responses remain in English with Chinese fields provided separately
- No external i18n library used (react-intl, next-intl, etc.) — lightweight custom implementation
- Dictionary files are statically imported (no lazy loading) for simplicity
